# Obsidian Writing Skill

生成干净、规范、对人可读且对 AI 可解析的 Obsidian Markdown 笔记。

本 skill 是 vault 中所有 .md 输出的**格式权威**。其他 skill 产出 .md 文件时，frontmatter 和正文格式必须遵守本规范。

## 依赖文件

执行写作任务前，根据需要读取以下文件：

| 文件 | 路径 | 用途 | 何时读取 |
| --- | --- | --- | --- |
| 模板 | `references/templates.md` | 各笔记类型的完整结构模板 | 生成笔记前，按类型读取对应模板 |
| 标签词表 | `references/tags.md` | 标签命名规则与可用标签清单 | 填写 frontmatter `tags` 字段时 |

如果依赖文件不可用，使用本文件中的精简骨架和核心标签规则作为 fallback。

## 核心原则

1. **CommonMark 对齐**：以 CommonMark 规范为基准，Obsidian 专有语法仅在必要时使用
2. **跨编辑器兼容**：输出在 Obsidian、Typora、GitHub 上均可正确渲染
3. **纯文本优先**：正文不使用 HTML 标签（`<br>`, `<div>`, `<span>` 等）
4. **中文优先**：正文使用中文，术语保留英文原文
5. **人类可读优先**：所有格式决策首先服务于人的阅读体验
6. **幂等性**：对同一篇笔记重复执行标准化，不应产生结构漂移或内容变更

## 兼容性速查

| 语法 | Obsidian | Typora | GitHub | 处理方式 |
| --- | --- | --- | --- | --- |
| `[[双链]]` | ✅ | ❌ 原始文本 | ❌ 原始文本 | 正常使用 |
| `> [!note]` Callout | ✅ | ⚠️ 普通引用块 | ⚠️ 普通引用块 | 正常使用，降级仍可读 |
| `==高亮==` | ✅ | ❌ | ❌ | **禁止使用**，用 `**粗体**` 替代 |
| `![[嵌入]]` | ✅ | ❌ | ❌ | 正常使用 |
| YAML frontmatter | ✅ | ✅ | ✅ | 正常使用 |

## YAML Frontmatter 规范

Frontmatter 是笔记的结构化身份证。所有 .md 输出必须遵守此 schema，这是 vault 级的合约——其他 skill 和清洗工具都依赖它。

### 标准字段（所有笔记必填）

```yaml
---
title: 笔记标题
date: 2025-02-25
tags: [标签1, 标签2, 标签3]
type: note
value: 1
summary: 一两句话概括笔记核心内容
---
```

### 字段定义

| 字段 | 必填 | 格式 | 说明 |
| --- | --- | --- | --- |
| `title` | 是 | 纯文本 | 与文件名保持一致 |
| `date` | 是 | `YYYY-MM-DD` | 笔记创建日期，后续编辑或清洗不修改此字段 |
| `tags` | 是 | YAML 行内数组 `[a, b, c]` | 分类标签，从 `references/tags.md` 选取 |
| `type` | 是 | 枚举值 | 笔记类型标识，见下方可选值 |
| `value` | 是 | 1-5 整数 | 价值评级，默认 `1`（未评级），见下方统一量表 |
| `summary` | 是 | 纯文本，1-2 句，50-100 字 | 自包含的核心内容概括 |

**tags 格式强制规定**：统一使用 YAML 行内数组格式 `[标签1, 标签2]`，不使用列表格式（`- 标签1`）。这确保清洗工具可以用统一的正则处理。

### 可选字段（按需添加，不堆砌）

| 字段 | 适用场景 | 格式 | 说明 |
| --- | --- | --- | --- |
| `source` | literature 类型 | 纯文本 | 来源信息 |
| `rating` | literature 类型 | 1-5 整数 | 个人评分 |
| `status` | article/analysis/project | 枚举值 | `draft` / `review` / `done` |

字段值包含冒号或特殊字符时用引号包裹。

### `type` 可选值

核心类型：`note`, `fleeting`, `literature`, `permanent`, `moc`, `project`, `article`, `analysis`, `log`, `meeting`, `person`, `weekly`

扩展类型：`conversation-distill`（对话沉淀笔记）

新增 type 须满足：有明确的结构模板、预计会产出 10+ 篇笔记。否则用 `note` + 标签区分。

### `value` 统一量表

此量表适用于所有笔记类型，衡量的是**笔记在知识体系中的复用价值**：

| 等级 | 含义 | 判定依据 |
| --- | --- | --- |
| 5 | 核心知识 | 反复引用的基础内容，或产生跨领域洞察 |
| 4 | 重要参考 | 对决策或学习有直接帮助，形成新的理解框架 |
| 3 | 有用内容 | 可复用的知识点或方法，偶尔查阅 |
| 2 | 一般记录 | 留作备查，小收获或修正小误解 |
| 1 | 未评级 | AI 生成时的默认值，待人工复盘后调整 |

### `summary` 书写规则

| 规则 | 说明 |
| --- | --- |
| 自包含 | 不依赖正文即可理解笔记主旨 |
| 信息密度 | 包含核心论点或结论，不是"本文讨论了 X" |
| 1-2 句话 | 控制在 50-100 字以内 |
| 避免双链 | summary 中不使用 `[[]]` 语法 |

好：`损失厌恶指人们对损失的敏感度约为同等收益的两倍，是前景理论的核心支柱，解释了风险决策中的非理性行为。`

差：`关于损失厌恶的笔记。`

### 标签规则

- 统一放在 frontmatter `tags` 字段，不在正文中使用行内标签
- 从 `references/tags.md` 选取，建议 2-8 个（按内容丰富度决定，宁少勿凑）
- 避免同义重复（`AI` 和 `人工智能` 只保留 `AI`）
- 大小写遵循品牌/官方规范（`iPhone`、`macOS`、`ChatGPT`）
- 技术缩写用大写（`AI`、`API`、`SQL`、`LLM`）
- 中文优先：能用中文表达的概念用中文（`提示工程` 而非 `prompt-engineering`）
- 英文保留条件：仅当概念没有通用中文译法时保留（`RAG`、`LLM`）
- 标签做宏观分类，双链做具体关联，两者互补不混淆

## 正文格式规范

### 标题层级

| 规则 | 说明 |
| --- | --- |
| `#` 仅用于文章主标题 | 每篇一个，与 frontmatter `title` 一致 |
| 正文章节从 `##` 开始 | 最深到 `####` |
| `#` 后必须有空格 | CommonMark 强制要求 |
| 标题前后各空一行 | markdownlint MD022 |
| 不跳级 | `##` 后不直接跟 `####`（MD001） |

### 排版规则

| 规则 | 示例 |
| --- | --- |
| 中英文间加空格 | `使用 Obsidian 管理 100 条笔记` |
| 中文全角标点 | ，。！？；：""（） |
| 英文/数字半角标点 | 正常英文标点 |
| 段落间空一行 | — |
| 不用尾部双空格换行 | 需要换行时用新段落 |
| 列表项间不空行 | 除非列表项包含多段落 |

### 强调与标记

| 语法 | 用途 | 约束 |
| --- | --- | --- |
| `**粗体**` | 关键概念首次出现、重点强调 | 一个段落最多 1-2 处 |
| `*斜体*` | 书名、外文原文、轻度强调 | — |
| `` `行内代码` `` | 代码、命令、文件名、技术术语 | — |
| `~~删除线~~` | 标记废弃内容 | — |
| `==高亮==` | **禁止使用** | 用 `**粗体**` 替代 |

### 列表

| 规则 | 说明 |
| --- | --- |
| 无序列表用 `-` | 不用 `*` 或 `+`（MD004） |
| 有序列表用 `1.` `2.` `3.` | — |
| 嵌套缩进 4 空格 | 跨平台兼容性最佳 |
| 任务列表 | `- [ ]` 未完成，`- [x]` 已完成 |
| 列表前后各空一行 | MD032 |

### 表格

```markdown
| 列1 | 列2 | 列3 |
| --- | --- | --- |
| 内容 | 内容 | 内容 |
```

超过 5 列或内容多时考虑用列表或分段替代。表格前后各空一行。

### 引用与 Callout

```markdown
> [!note] 标题
> 内容
```

每篇不超过 2-3 个 Callout。折叠用 `-` 号：`> [!note]- 可折叠标题`。

### 代码块

标注语言（MD040），前后各空一行（MD031）。

### 分隔线

使用 `***`（避免与 YAML `---` 混淆，MD035），单独一行，前后各空一行。

### 修订记录

每篇笔记末尾附带可折叠修订记录：

```markdown
> [!abstract]- 修订记录
> **2025-04-02** 修正损失厌恶系数：原文"3倍"改为"约2倍"
> **2025-02-25** 初稿
```

时间倒序，每条一行含修改理由。纯格式修正不记录，AI 修改必须记录。

## 双链策略

### 该链与不该链

| 该加双链 | 不该加双链 |
| --- | --- |
| 人名、书名/作品 | 过于宽泛的概念（`[[科学]]`） |
| 独立概念/理论 | 常识性词汇（`[[互联网]]`） |
| 项目名、工具/平台 | 一次性提及的细节 |
| MOC 入口 | — |

### 双链规则

| 规则 | 说明 |
| --- | --- |
| 首次提及时链接 | 同一概念在一篇中只链一次（MOC 例外） |
| 自然融入句子 | 双链是句子的有机部分 |
| 文末不堆链接 | 链接出现在正文讨论的位置 |
| 大胆链接，不怕红链 | 红链是知识空白信号 |

### 命名规范

| 类型 | 规范 | 示例 |
| --- | --- | --- |
| 中文人名 | 直接使用 | `[[张三丰]]` |
| 外文人名 | 中文译名 | `[[卡尔·波普尔]]` |
| 书名（有读书笔记） | 加后缀 | `[[《思考，快与慢》读书笔记]]` |
| 书名（仅提及） | 书名号 | `[[《思考，快与慢》]]` |
| 概念/理论 | 最常见中文名称 | `[[损失厌恶]]` |
| MOC | `MOC-` 前缀 | `[[MOC-认知科学]]` |
| 别名 | 管道符 | `[[卡尔·波普尔\|波普尔]]` |

### 链接密度参考

| 笔记类型 | 建议密度 |
| --- | --- |
| fleeting | 1-3 个 |
| literature | 5-10 个 |
| permanent | 3-6 个 |
| moc | 不限 |
| article/analysis | 5-15 个 |
| conversation-distill | 3-8 个 |
| project | 3-8 个 |
| log | 1-5 个 |

## 文件命名

| 规则 | 说明 |
| --- | --- |
| 文件名 = 标题 = 双链名 | 三者一致 |
| 中文简体 | 简洁明确 |
| 不加日期前缀 | 日期交给 frontmatter |
| 中英文/数字间加空格 | `2026 年 Q1 读书清单` |
| MOC 加前缀 | `MOC-科学哲学` |

**笔记原子性**：一条笔记一个主题。超过 1000 字且多子主题时考虑拆分。

## 笔记类型判断

生成笔记前先判断类型，再从 `references/templates.md` 读取对应模板。

| 用户意图 | type 值 | 建议目录 |
| --- | --- | --- |
| "帮我记一下…" "突然想到…" | `fleeting` | `0-Inbox` |
| "读了这本书/文章…" "总结一下…" | `literature` | `3-Resources` |
| "我的理解是…" "关于 X 的思考" | `permanent` | `3-Resources` |
| "帮我整理这个主题的笔记" | `moc` | `3-Resources` |
| "写一篇文章/报告/分析" | `article` / `analysis` | `2-Areas` 或 `1-Projects` |
| "这个项目的情况…" | `project` | `1-Projects` |
| "今天做了什么" / "本周回顾" | `log` / `weekly` | `5-Daily` |
| "今天开会讨论了…" | `meeting` | `1-Projects` |
| "关于这个人…" | `person` | `2-Areas` |
| "沉淀一下" / "提取笔记" | `conversation-distill` | `0-Inbox` |
| 其他 | `note` | `2-Areas` 或 `3-Resources` |

建议目录在对话中标注，不写入 .md 文件。

## 输出流程

1. 判断笔记类型
2. 读取 `references/templates.md` 对应模板（不可用时使用模板精简骨架）
3. 填写 frontmatter：所有必填字段 + 按需可选字段，`value` 默认 1
4. 从 `references/tags.md` 选取标签
5. 按正文格式规范填充内容，自然嵌入双链
6. 末尾添加修订记录
7. 执行格式自检
8. 在对话中标注建议存放目录

### 格式自检清单

| 检查项 | 对应规则 |
| --- | --- |
| frontmatter 完整（title/date/tags/type/value/summary） | 字段定义表 |
| tags 使用行内数组格式 `[a, b]` | tags 格式强制规定 |
| 标题前后有空行 | MD022 |
| 标题不跳级 | MD001 |
| 列表缩进 4 空格 | 跨平台兼容 |
| 代码块前后有空行 | MD031 |
| 列表前后有空行 | MD032 |
| 分隔线统一 `***` | MD035 |
| 中英文间距正确 | 排版规则 |
| 未使用 `==高亮==` | 兼容性要求 |
| 修订记录在末尾 | 修订记录规范 |
| 无 HTML 标签 | 纯文本优先 |
| 双链首次提及 | 双链规则 |

## 模板精简骨架（Fallback）

当 `references/templates.md` 不可用时使用：

- **fleeting**：`## 想法` → `## 触发` → `## 下一步`
- **literature**：`## 核心论点` → `## 关键内容` → `## 个人收获` → `## 疑问与待探索`
- **permanent**：无二级标题，直接正文 3-5 段，标题是完整陈述句
- **moc**：`## 概述` → `## 核心概念` → `## 关键问题` → `## 资源与来源` → `## 待探索`
- **article**：`## 引言` → `## 主体` → `## 结语`
- **analysis**：`## 摘要` → `## 背景` → `## 方法/框架` → `## 分析` → `## 结论与建议` → `## 局限性`
- **project**：`## 目标` → `## 当前状态` → `## 关键决策` → `## 待办` → `## 参考资料`
- **log**：`## 今日完成` → `## 收获与思考` → `## 明日计划`
- **weekly**：`## 本周亮点` → `## 完成事项` → `## 未完成与阻碍` → `## 收获与反思` → `## 下周重点`
- **meeting**：`## 基本信息` → `## 议题与讨论` → `## 决策` → `## 后续行动`
- **person**：`## 基本信息` → `## 关键印象` → `## 交流记录` → `## 关联`
- **conversation-distill**：`## 关键结论` → `## 新知识/认知更新` → `## 开放问题` → `## 待办/下一步` → `## 关联笔记`（前置 `## 原始素材/上下文` 仅在有实质内容时使用）
- **note**：结构自由
